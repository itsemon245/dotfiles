Before ^GYITJjRB

- 1 reverb instance
- This has 1024 File Descriptor cap because of php is compiled with that limit
- So more or less ~1000 clients can connect the the server concurrently before we hit the FD limit ^qdhcXkWg

Solutions ^O7sfuKH7

1. we can recompile the php binary to increase the FD limit for each process to a higher number
2. we can deploy multiple instances of `REVERB` to handle more clients ^XhK5MVbB

The solution I have chosen ^FDUd1hYN

I have chosen the 2nd solution, and here are the reasons:

1. If we recompile php with higher number of FD limit there could be unexpected side-effects which can bring technical debt down the road.

2. Even if we recompile php we would have to do that again in the future as our user-base grows

3. Deploying multiple instances of the websocket server makes more sense and is more scalable and we can scale things up and down based on the users activity ^JpCwVOhm

Context ^CEm2xLlB

The Plan ^0G0Qugbh

The plan is to run multiple instances of reverb no matter the server.

Load balance them using nginx

Let them fire events to a redis channel so every clients can receive it regardless of the connected server for a client ^3g3w64gx

1. Subscribes to the events from the reverb server

2. Publishes them to all instances of the reverb server so every client can receive the updates ^KtsK6DnC

Maintainability and Resources Usage Concerns ^T8SZfWvy

1. The more instance we have the more concurrent clients we can serve.

2. Using pm2 we can manage and limit the memory usage for each process. default is 256MB which should suffice for our case.

3. Redis would take some amount of memory and we can get the estimate after the implementation is complete.

4. Now we have to dispatch more instances of reverb, add them to nginx upstream and reload nginx to scale whenever we need. We can automate this with a bash script so it can be a single line command.
